neo-notion-cli

Agent-first Notion API CLI with normalized, context-efficient output. Covers the full Notion API surface — pages, databases, blocks, comments, search, webhooks — in a single binary.

View on GitHub How It Works
The Problem

Notion's API burns through
agent context

The Notion API returns deeply nested JSON. A single page from a query is ~12 KB of annotations, type wrappers, empty objects, and request metadata. An agent running a handful of queries can exhaust its context window on boilerplate before it even starts reasoning.

# Raw Notion API response: 12,615 bytes
 
{
  "properties": {
    "Title": { "title": [{ "plain_text": "My Page",
      "annotations": { "bold": false, "italic": false,
        "strikethrough": false, "underline": false,
        "code": false, "color": "default" },
      "type": "text", "text": { "content": "My Page",
        "link": null }, "href": null }] },
 
  // ... 200+ more lines of nesting
# neo-notion-cli: same data, 3.5x smaller
 
$ notion data-sources query --data-source-id $DS --page-size 1
 
{
  "title": "My Page",
  "Status": "Done",
  "Tags": ["A", "B"],
  "Due": "2025-12-21",
  "Done": true
}
 
# 3,582 bytes — all the signal, none of the noise
The Solution

A normalization layer
for context-efficient output

neo-notion-cli compresses API responses to their semantic core. Properties become scalar values, rich text becomes plain strings, annotations and metadata are stripped. Agents get the same data in a fraction of the tokens.

Capabilities

Full Notion API
in a single binary

Search & Query

Search by title across all pages and data sources. Query data sources with filters, sorts, and pagination. Built-in jq-like field selection with --select.

notion search --query "Q1 Planning" --select '.results[] | {title}'

Pages & Blocks

Create, read, update, and move pages. Full block tree traversal with --recursive. Read and write page content as markdown via dedicated endpoints.

notion pages markdown get --page-id <id>

File Uploads

Multi-part file upload support. Create upload sessions, send file parts, and complete uploads — all from the command line. Supports piping from stdin.

notion file-uploads send --file-upload-id <id> --file report.pdf

Normalized Output

Responses are compressed to their semantic core by default. Properties become scalars, annotations are stripped, envelopes are flattened. 3.5x smaller on average.

notion users me --full # skip normalization

OAuth & Webhooks

Full OAuth 2.0 flow with browser-based login. Token create, refresh, introspect, and revoke. Local webhook listener with automatic verification challenge handling.

notion auth login --oauth --client-id <id>

Flexible Input

All mutating commands accept bodies three ways: inline JSON, from a file, or piped from stdin. Auto-pagination, retry with backoff, and idempotency keys built in.

cat page.json | notion pages create --stdin
Architecture

Built with precision

Clean separation between CLI commands, HTTP client, and the normalization layer. Every command follows the same pattern: parse flags, call the typed API client, normalize, format.

# Architecture
 
cmd/ # Cobra commands (12 resource groups)
auth/ # login, token create/refresh/introspect/revoke
pages/ # create, get, update, move, markdown *
blocks/ # append, get, children (--recursive), delete
datasources/ # create, get, update, query, templates
comments/ # create, get, list
webhooks/ # listen (local server), events
 
internal/
httpx/ # Retryable HTTP client + Notion headers
notion/ # Typed API client (all endpoints) + jq select
render/ # THE CORE — normalization + output formatting
config/ # Layered config: flags > env > file
agents/ # Body loading (--body, --body-file, --stdin)
Why This Matters

Context is the bottleneck
for agents

Every token matters when you're running an AI agent. Feeding raw API responses into an agent's context window is like handing someone a phone book when they asked for a phone number.

neo-notion-cli solves this at the tool level. Instead of asking agents to parse and discard boilerplate, the CLI delivers only the semantic content — the titles, statuses, dates, and relations that agents actually need to reason about.
