Expose any local app to the internet in one command — secured with Cloudflare Tunnel + Access. Built for AI agents and developers.

The Problem

Exposing local apps
is a 5-step nightmare

To get a local app publicly accessible with authentication, you normally need to: configure a tunnel, create a DNS record, set up an access policy, manage credentials, and wire it all together. For agents deploying preview environments, this is a non-starter.

# Without flare-cli: the manual way
1. Create Cloudflare tunnel via API
2. Write ingress config YAML
3. Create CNAME DNS record
4. Create Access application + allow policy
5. Start cloudflared with correct flags
 
# And when you're done:
# Manually delete tunnel, DNS, Access app
# Hope you didn't leave anything behind
# With flare-cli: one command
$ flare serve builtin:static --path ./dist --subdomain myapp --auth otp
 
✓ Origin ready at http://127.0.0.1:49152
✓ Tunnel created: flare-a1b2c3d4
✓ DNS record created for myapp.example.com
✓ Access application created (auth: otp)
✓ cloudflared running (PID 12345)
 
Session Active
  Hostname: https://myapp.example.com
  Auth:     otp
The Solution

One command from local
to live

flare-cli orchestrates the entire pipeline: start the origin, create a tunnel, set up DNS, apply access policies, and connect through Cloudflare's edge. On teardown, everything is cleaned up automatically — no leaked resources.

Origin Types

Expose anything
with flare-cli

Static Files

Serve static files from a directory with SPA routing support, custom cache headers, and configurable index files. Perfect for React, Vue, or plain HTML builds.

flare serve builtin:static --path ./dist --spa

Local Services

Proxy an already-running local HTTP service, or launch a command and expose its port. Health checks and startup timeout built in.

flare serve local:http --url http://localhost:3000

Docker Containers

Run a Docker image or a Compose service and expose it publicly. Supports bind mounts, environment variables, entrypoint overrides, and auto-cleanup.

flare serve docker:container --image nginx:alpine --container-port 80

File Browser

Share a directory with a built-in file browser UI. Enable uploads, downloads, renames, and deletion per session. Control access with Cloudflare's auth layer.

flare serve builtin:file-browser --path ~/shared --allow-upload

Access Control

Every session is protected by Cloudflare Access. Choose OTP, identity provider, mTLS, or service tokens. Restrict by email or domain. Update policies on running sessions.

flare serve ... --auth otp --allow-domain company.com

Auto-Rollback

If any step fails during provisioning, all prior resources are rolled back automatically. On teardown, tunnel, DNS, and Access app are cleaned up. Zero leaked resources.

flare close <session-id>
Pipeline

How it works

A 7-step pipeline that creates everything needed to go from local to live — and tears it all down in reverse when you're done.

# flare serve pipeline
 
1. Start origin # Launch local server / container / proxy
2. Create tunnel # Cloudflare API: remotely-managed tunnel
3. Configure ingress # Set routing: hostname → origin
4. Create DNS record # CNAME: subdomain.domain → tunnel
5. Create Access app # App + allow policy (OTP/IdP/mTLS)
6. Start cloudflared # Connect tunnel to Cloudflare edge
7. Session active # Wait for Ctrl+C or TTL expiry
 
# flare close: reverse teardown
Stop cloudflared → Delete Access → Delete DNS → Delete tunnel
Why This Matters

Agents need to
deploy things

AI agents increasingly need to expose services — preview environments for PRs, webhook receivers, demo apps, file shares. Doing this manually through cloud dashboards isn't an option when an agent is the one making the decision.

flare-cli gives agents a single command with JSON output, stable session IDs, and idempotency keys for safe retries. An agent can create a preview environment, check its status, and tear it down — all programmatically.

And because everything runs through Cloudflare's infrastructure, it's secured by default. No exposed ports, no open firewall rules, no unprotected endpoints.
